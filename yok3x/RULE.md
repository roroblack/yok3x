# RULE.md — yok3x 멀티 에이전트 운영·개발 규칙

이 문서는 yok3x 프로그램을 **만들고 쓸 때** 지켜야 할 규칙이다. 코드·리포트·에이전트 작업 모두 여기에 따른다.

---

## 1. 정체성 — 코딩용 멀티 에이전트 도구

- 목적: Claude·Codex·Gemini CLI를 묶어 **구독 한도를 지키며** 코딩 루프(구현→리뷰→수정)를 돌린다.
- YouTube/콘텐츠용이 아니라 **코딩용**이다. 워커·태스크·라우팅은 코딩 워크플로우 기준.
- 의존성 0(파이썬 표준 라이브러리만). 새 의존성 추가 금지.
- Windows 우선 지원(개발자 실사용 환경). 모든 변경은 Windows에서 검증한다.

## 2. 한도는 무조건 지킨다 (요금 가드)

- 실측 한도가 있으면 그것이 기준이다: **codex = app-server 라이브 실측**, claude = transcript 롤링 추정(+`calibrate` 보정), gemini = 원장.
- 매 워커 호출 전 가드 점검. `hard_ratio` 도달 시 루프 자동 정지. 실측 실패 시 `on_probe_failure` 정책(기본 ledger 폴백).
- 정확한 수치를 모르면 **지어내지 말 것**. claude 토큰 상한은 `plan` 프리셋(근사) 또는 `calibrate`(정확)로만 설정.

## 3. yok3x 기법 (프로그램을 만들 때 적용)

코드를 생성/수정하는 워커에는 **계획 → 구현 → 자가검증** 구조를 강제한다(`orchestrator.YOK3X_TECHNIQUE`).
폭주 방지 4중 브레이크를 항상 건다:

1. **단계 분할** — 한 번에 다 하지 않고 스텝 단위로 실행·로그.
2. **사람 승인** — 게이트(y/N/q). `--auto`는 신뢰된 루프에서만.
3. **항상 검증** — 검수 워커(producer-reviewer) + 체크리스트.
4. **예산 제한** — 요금 가드.

## 4. 할루시네이션 방지 (필수)

- 모든 워커 프롬프트에 사실성 규칙 주입(`orchestrator.ANTI_HALLUCINATION`): 추측을 사실로 쓰지 말 것, 모르면 '모름', 존재하지 않는 파일·함수·API·플래그 날조 금지, 근거 제시.
- 검수 워커는 환각/날조를 명시 지적(`REVIEW_GUARD`).
- 체크리스트가 근거 없는 과잉 확신 표현을 자동 표시.
- **에이전트(및 이 프로그램을 만드는 AI)도 동일 규칙을 따른다**: 파일·함수·수치를 주장하기 전에 실제로 확인(Read/실행)하고, 검증 못 한 것은 "미확인"으로 표시한다. 렌더/실행으로 확인 가능한 것은 코드만 보고 단정하지 않는다.

## 5. 검증 우선 (코드만 보고 판단 금지)

- 동작·수치·렌더는 **실제로 실행/확인**하고 보고한다. 실패는 실패대로, 스킵은 스킵대로 정직하게.
- "될 것이다"가 아니라 "실행해서 됐다/안 됐다"로 말한다.

## 5.5 폴백·하드코딩 최소화 (필수)

- **폴백(fallback)과 하드코딩은 꼭 필요한 곳에만** 적용한다. "일단 돌아가게" 식 임시 폴백·매직넘버 금지.
- 허용 기준:
  - 폴백: 외부 자원(CLI·파일·네트워크) 실패 시 **명시적 열화 경로**가 사용자에게 표시되는 경우만 (예: 실측 실패 → `⚠stale`/`원장` 배지). 조용히 값을 지어내는 폴백 금지.
  - 하드코딩: 프로토콜 상수(API 필드명·창 크기 300/10080분 등)·안전 상한(타임아웃)은 허용. **사용자 환경마다 다른 값**(예산·한도·경로·모델명)은 반드시 설정(yok3x.json)으로 뺀다.
- **적용할 때는 별도 리포트를 만들어 제출한다**: 어디에 · 왜 필요한지 · 실패 시 어떻게 표시되는지 · 설정으로 뺄 수 없는 이유. 리포트 없는 폴백/하드코딩 추가 금지.
- 기존 폴백/하드코딩 현황은 최신 `reports/*-fallback-hardcoding-audit-*.md` 레지스트리에서 추적한다.

## 6. 리포트 파일 네이밍 규칙

`reports/` 아래 리포트 파일명은 **[버전]-[짧은 작업명]-[날짜]-[시간]** 형식:

```
v2.3.0-gui-prototype-2026-07-04-1336.md
v2.3.0-coding-fit-review-2026-07-04-1340.md
```

문서 맨 위 헤더에도 동일하게 `버전 · 작업 · 날짜 시간`을 적는다.

## 7. 변경 이력 (HISTORY.md)

- 코드·기능·문서를 바꾸면 **매번 [HISTORY.md](HISTORY.md)에 한 줄 이상 기록**한다: 버전, 날짜/시간, 변경 요약.
- 릴리스(zip)를 만들 때 버전을 올리고 HISTORY에 릴리스 항목을 남긴다.

## 8. 릴리스 · 버전 보존

- 정본 소스: `yok3x/`. 배포물: `release/yok3x-vX.Y.Z.zip`.
- zip 만들기 전 `__pycache__`·런타임 산출물(`.yok3x/`·생성된 `yok3x.json` 등) 제거.
- Windows 스모크 테스트(`init/setup/run/gui/limits`) 통과 후 패키징.
- **이전 버전 zip을 삭제하지 않는다.** 새 릴리스를 만들면 이전 릴리스 zip을 `backup/versions/`로 옮기고 `backup/versions/VERSIONS.md`에 항목을 추가한다(내부 `__version__`으로 검증). 리브랜딩 등 대규모 변경 전에는 정본 폴더 전체를 `backup/`에 스냅샷한다.
