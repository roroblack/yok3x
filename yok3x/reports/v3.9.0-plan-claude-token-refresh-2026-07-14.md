# v3.9.0 · 계획서 — claude OAuth 토큰 자체 갱신 · 2026-07-14

## 목표
claude 실측(5h/7d)이 **토큰 만료(~1시간)로 반복적으로 끊기던** 문제를 해소. yok3x가 만료 임박 시
`refresh_token`으로 access_token을 **표준 OAuth 갱신** → 실측이 무중단으로 유지(하루 페이싱 등도 실측 기반).
(분석: `v3.x-assessment-claude-token-refresh-2026-07-14.md` — 차단 대상 아님, near-expiry 갱신+백오프가 정답.)

## 현재 구조(근거)
- `~/.claude/.credentials.json` → `claudeAiOauth`: `accessToken`·**`refreshToken`**·`expiresAt`(ms)·scopes·
  subscriptionType. `_claude_oauth_token`이 accessToken을 읽고, `expiresAt`이 지났으면 None(만료) 반환.
- 만료 시 실측 실패 → stale/추정/원장 폴백(무중단이나 실측 %는 사라짐).

## 설계

### 1) 갱신 트리거 — near-expiry (고정 타이머 아님)
- `_claude_oauth_token`이 토큰을 읽을 때, `expiresAt - now < refresh_margin_sec`(기본 300초)면 **갱신 시도**.
- 이미 만료됐어도 refresh_token 있으면 갱신 시도. 자연히 ~1시간에 1번 → 공식 클라이언트와 동일 주기.

### 2) 갱신 요청 (표준 OAuth refresh)
- `POST <token_url>` `{grant_type: "refresh_token", refresh_token, client_id}` (Content-Type: json).
  - `token_url`·`client_id`는 **config로** 뺀다(하드코딩 금지, §5.5). 기본값은 구현 시 실검증으로 확정.
- 응답 `{access_token, refresh_token?, expires_in}` → **원자적으로 `.credentials.json` 갱신**
  (temp→replace). refresh_token이 회전(rotate)되면 새 값 저장.

### 3) 폭주 방지·실패 안전
- `min_refresh_interval_sec`(기본 60)로 과다 갱신 차단 + 실패 시 **지수 백오프(+소량 지터)** — 재시도 폭주 방지.
- 갱신 실패해도 기존 폴백(stale/추정/원장) 유지 → **무중단(fail-safe)**. 갱신은 '보너스'.
- 프로세스 내 캐시로 동시 다중 갱신 억제.

### 4) 동시성·안전
- Claude Code 자체도 이 파일을 갱신하므로 **읽기→판단→쓰기**를 원자적으로: 쓰기 직전 파일을 **재로드**해
  더 새 토큰이 있으면 그걸 채택(우리 것으로 덮어쓰지 않음). 손상 방지 위해 temp→replace.
- 자격 처리: 사용자 **자기 토큰**을 표준 플로우로 갱신(폼 입력 아님). 로컬 전용. RULE 금지행위 아님
  (계정 인증/비번 입력이 아니라 기존 refresh_token 사용).

### 5) 설정 (opt-in vs 기본)
- `limits.claude.auto_refresh`(기본 **true** — 사용자가 원한 동작이고 표준), `refresh_margin_sec`(300),
  `min_refresh_interval_sec`(60), `token_url`, `client_id`. 끄면 현행(폴백)대로.

## ⚠ 라이브 미검증(의도적) — 사용자 확인 필요
- **실 credentials로 갱신 라이브 테스트를 하지 않았다.** refresh_token이 회전(rotate)되면 실제 Claude Code
  인증까지 소비/변경될 수 있어 위험(codex도 지적). 목으로 로직만 검증.
- 따라서 `token_url`·`client_id` 기본값은 **Claude Code 공개 OAuth 값(미검증)**. 켜기 전 확인 권장.
- **검증 방법**: `auto_refresh: true`로 켠 뒤 토큰 만료 임박 시 실측이 유지되면 성공. 실패해도 fail-safe
  (기존 폴백, 자격 안 씀). 4xx면 회로차단+재로그인 안내. → 원하면 사용자 승인 하에 1회 라이브 검증.

## 결정 필요
1. `auto_refresh` 기본값: **true 추천**(원하는 동작). 보수적으로 가려면 false(opt-in).
2. `token_url`·`client_id` 기본값: 구현 시 **실검증**으로 확정(사용자 refresh_token으로 1회 테스트). 값은
   config로 override 가능하게.

## codex 리뷰 반영 (SCORE 7/10, 위험도 중간 — 보수적 적용)
- **회전토큰 잠김 방지**: 새 토큰을 **즉시** 저장(temp fsync→replace), **기존 토큰은 백업 보존**. 응답 후
  저장 전 크래시 창은 완전 제거 불가 → 기존 access_token 폴백 경로 유지.
- **`invalid_grant`(4xx) = 즉시 회로차단**: 반복 갱신 금지, 기존 폴백 + 재인증 안내. 4xx 영구오류는
  재시도 안 함. **429/5xx/네트워크 오류만** 백오프 재시도. 최대 연속 실패/쿨다운.
- **동시쓰기(Claude Code와 공유)**: 쓰기 직전 재로드→더 새 토큰이면 채택(우리 것으로 안 덮음)→필드
  병합→temp→replace. 파일락 가능하면 사용. 우리가 갱신한 새 토큰을 **파일에 되써서** Claude Code도
  같은 파일로 동기화되게(캐시 불일치 최소화).
- **하드코딩 회피**: `token_url`·`client_id`는 config(또는 credentials 메타데이터)에서. **출처 불명/불일치면
  추측하지 말고 갱신 비활성**(기존 폴백).
- **기본값 결정**: 위험 중간이라 codex '보수적' 권고 반영 — `auto_refresh` **기본 false(opt-in)**로 하되
  켜기 쉽게. 사용자가 명시적으로 켜면 동작(원한 기능). 켜도 모든 실패는 fail-safe.
