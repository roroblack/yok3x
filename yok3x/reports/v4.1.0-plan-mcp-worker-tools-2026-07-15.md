# v4.1.0 · 계획서 — MCP 워커 도구 (Model Context Protocol) · 2026-07-15

## 목표 (분석: v3.x-analysis-langgraph-mcp)
현재 워커는 `--disallowedTools`로 파일 접근이 막힌 **텍스트 생산자**. MCP 도구 서버(filesystem·git·DB·web)를
붙여 워커가 **가드·승인 아래 실제 행동**(파일 읽기·쿼리·검색)을 근거로 작업하게 한다. `backends.json`에
`type: mcp` 자리표시자 이미 있음. 의존성0(JSON-RPC는 stdlib subprocess+json).

## 현재 구조(근거)
- 워커 = CLI 서브프로세스(claude/codex/gemini) — **각자 이미 MCP를 지원**(claude/codex는 MCP 서버 등록 가능).
- claude 기본 명령에 `--disallowedTools Bash,Edit,Write,Read,…`로 도구 차단(안전=텍스트만).

## 설계 — 두 형태, 우선 (a1)

### (a1) yok3x가 워커 CLI에 MCP 서버 '설정을 전달' — **1차 구현(가장 단순·안전)**
- yok3x가 `mcp_servers` 설정(이름→{command/url})을 워커 CLI의 MCP 설정으로 넘긴다
  (claude `--mcp-config`, codex의 MCP 설정 등 — backends.json에 per-backend `mcp_arg` 템플릿).
- 특정 워커/작업에만 도구 허용: `--disallowedTools`를 완화하고 **허용 도구를 화이트리스트**로.
- 워커가 자기 런타임에서 MCP 도구 사용 → 결과가 산출물에 반영. yok3x는 설정 전달·게이트만.

### (a2) yok3x가 MCP 클라이언트로 직접 도구 호출 — 후속(복잡)
- yok3x가 JSON-RPC로 MCP 서버에 tools/list·tools/call → 결과를 프롬프트에 주입. 서버 수명주기 관리.
- (a1)로 충분하면 미룬다(과설계 지양).

### (b) yok3x를 MCP 서버로 노출 — 별도 후속
- 오케스트레이션(run·knot query)을 MCP 서버로 열어 Claude Code 등이 도구로 호출. 조립성.

## 보안 (핵심 — RULE 통합 필수)
- 파일/실행 MCP는 강력 → **현재 '텍스트 생산자=안전' 설계를 의도적으로 완화**. 반드시:
  - **opt-in**(기본 도구 차단 유지). 작업 spec/설정에서 명시적으로 켠 워커만.
  - **승인 게이트 연동**: 도구 사용 워커는 auto_approve=false 권장. RULE 액션 분류(파일쓰기·삭제 등) 준수.
  - **화이트리스트**: 허용 MCP 서버·도구를 설정에 명시. 임의 서버 금지.
  - 워크스페이스 격리(`_isolated_cwd`/workdir) 유지 — 도구 접근 범위 제한.

- **화이트리스트+승인만으론 부족**: 도구명뿐 아니라 **인자·경로·호스트·작업범위 검증**, 위험 등급, 취소/타임아웃.

  호출/시간/비용 한도·네트워크 제한**.
- **누락 보완**: 호출 예산, 인자 정책, 격리 경계, **감사 로그**, **실패 시 기본 거부(fail-closed)**.
