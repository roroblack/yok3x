# 분석 — LangGraph에서 가져올 아이디어 · MCP 도입 이점 · 2026-07-14

yok3x 철학: **프레임워크는 안 들이고 좋은 아이디어만 의존성0으로 이식**(ARIS·LightRAG·Mem0 때와 동일).
그 관점에서 LangGraph·MCP를 평가한다.

---

## 1. LangGraph에서 가져올 아이디어 (프레임워크 도입 ✗, 아이디어 ✅)

yok3x가 **이미 가진 것**: 3패턴(producer-reviewer/pipeline/fanout), 스톨 감지, 승인 게이트, 적응형 열화
(P1 모델다운·P2 폴오버·P3 오프라인), status.json 단계 로그, 요금 가드. → LangGraph의 '순환·스트리밍·
휴먼인더루프' 상당부분은 이미 커버됨.

**LangGraph의 차별점 중 yok3x에 없는 것 = 가져올 후보:**

### ⭐ A. 조건부 라우팅(conditional edges) — 가치 높음
- LangGraph 핵심: 노드 출력에 따라 **다음 단계를 동적으로** 결정("if 조건 → node X").
- yok3x는 패턴이 **정적**(고정 순서). 가져오면: task spec에 **조건부 전이** 정의 —
  예: "2라운드 후 SCORE<5면 더 강한 모델/다른 워커로 에스컬레이션", "리뷰어가 보안 결함 지적 시
  보안 특화 워커로 라우팅". 이미 있는 스톨감지·열화를 **일반화**하는 자연스러운 확장.
- 의존성0: task spec에 `on: {score_lt: 5 → escalate}` 같은 규칙 필드 + 오케스트레이터 분기. 신규 의존성 0.

### ⭐ B. 체크포인트 재개(resumable checkpoints) — 가치 중상
- LangGraph: 각 단계 상태를 저장해 **중단→재개/되감기**.
- yok3x는 status.json·step 파일을 **쓰지만 mid-run 재개는 없다**. 가드 stop·승인 대기·크래시로 멈춘
  런을 **N단계부터 이어서** 못 한다. 특히 방금 만든 pace-approve·guard-stop과 궁합 —
  "정지된 런을 승인 후 이어서". `yok3x resume <run_id>`로 이식 가능(이미 step 산출물이 있음).
- 의존성0: 기존 status.json/step 파일에서 상태 복원 + 오케스트레이터 재진입점. 신규 의존성 0.

### 낮음/불필요
- 휴먼인더루프 인터럽트: yok3x 승인 게이트·pace-approve로 **이미 커버**.
- 순환/루프: producer-reviewer 루프+스톨감지로 커버. 스트리밍: status.json 폴링으로 커버.
- 서브그래프(중첩 워크플로): 나중 후보(지금 불필요).

**결론(LangGraph):** 프레임워크는 배제하되 **①조건부 라우팅 ②체크포인트 재개** 두 아이디어는 이식 가치
있음. 둘 다 의존성0. 나머지는 이미 있음.

## 2. MCP 도입 이점 분석 (LangGraph보다 적합)

MCP(Model Context Protocol) = LLM↔도구/데이터를 잇는 **표준 프로토콜**(JSON-RPC, stdio/HTTP). `backends.json`
에 이미 `type: mcp` **자리표시자**가 있음(도입 염두). LangGraph와 달리 **의존성0으로 구현 가능**(표준
subprocess+json), 그리고 **철학(설정 주도)과 정합**.

### 이점
1. **워커에게 '실제 행동' 부여(가장 큰 이점).** 현재 yok3x 워커는 `--disallowedTools`로 파일 접근이 막힌
   **텍스트 생산자**(코드를 텍스트로만 출력). MCP 도구 서버(filesystem·git·DB·web)를 붙이면 워커가 가드·
   승인 아래 **실제 파일 읽기·쿼리·검색**을 근거로 작업. "워커가 파일을 못 만진다"는 실제 한계를 해소.
2. **설정만으로 확장(코드 0).** 통합마다 어댑터를 짜는 대신 MCP 서버를 **설정으로 연결**. yok3x의 설정
   주도·의존성0 철학과 정확히 맞음(MCP는 JSON-RPC라 stdlib로 클라이언트 구현 가능).
3. **생태계 재사용.** GitHub·Slack·Postgres·브라우저 등 **기존 MCP 서버**를 그대로 활용 — 바퀴 재발명 회피.
4. **yok3x를 MCP 서버로 노출.** 오케스트레이션(producer-reviewer 실행·knot 조회)을 **MCP 서버로** 열면
   Claude Code 등 외부 에이전트가 yok3x를 **도구로 호출**. yok3x가 더 큰 에이전트 시스템에 조립됨.
5. **워커 CLI의 기존 MCP 활용.** claude/codex CLI는 **이미 MCP를 지원** — yok3x가 MCP 서버 설정을
   워커에 넘기면 워커가 자기 런타임에서 도구 사용.
6. **표준·미래 대비.** MCP가 사실상 표준화 중 — 상호운용성 확보.

### 비용·주의
- **보안이 핵심.** 파일시스템/실행 MCP를 붙이면 워커가 강력해짐 → yok3x **승인 게이트·RULE 액션 분류**와
  반드시 통합(현재의 '텍스트 생산자=안전' 설계를 의도적으로 완화하는 것이라 게이트 필수).
- MCP 서버 **수명주기**(spawn·헬스·auth) 관리 복잡도. 자리표시자만 있고 실구현은 없음.
- 의존성0 유지하려면 SDK 대신 **직접 JSON-RPC 클라이언트**(subprocess/HTTP) 구현.

### 두 가지 도입 형태
- **(a) yok3x = MCP 클라이언트**: 워커가 MCP 도구 사용(파일·DB·web). ← '행동하는 워커' 이점.
- **(b) yok3x = MCP 서버**: 오케스트레이션을 외부에 도구로 노출. ← 조립성.

**결론(MCP):** LangGraph보다 **적합도 높음** — 설정 주도·의존성0 가능·실제 한계(워커 행동 불가) 해소.
승인 게이트 통합을 전제로 **계획서 가치 있음**. (a) 클라이언트부터가 임팩트 큼.
