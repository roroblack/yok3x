# v3.6.0 · 계획서 — 주간 쿼터의 '하루 페이싱' 한도(일일 소비 캡) · 2026-07-13

## 목표
주간(7d) 구독 쿼터를 **하루 단위로 페이싱**한다. "하루에 주간 쿼터의 N%까지만 써라"를 설정하고,
그 한도에 닿으면 **(a) 경고만** 하거나 **(b) 일시정지 후 사용자가 승인해야 재개**하도록 옵션을 준다.
→ 주초에 몰아 써서 주 후반 고갈되는 것을 막고, 소비 속도를 사용자가 통제.

## 현재 구조(근거)
- `usage.check_backend` → `GuardVerdict(level: ok/warn/stop)`. 임계는 `guard.soft_ratio`(경고)/`hard_ratio`(정지).
- 실측: `limits.probe`가 `windows`(5h·7d의 `used_percent`, =주간 쿼터 대비 %) 반환. `ratio()`=최대 창.
- 원장: `today_totals`가 `usage.jsonl`에서 **오늘(자정 리셋)** 백엔드별 usd/tokens/calls 집계.
- 즉 "주간 쿼터 %"는 이미 7d `used_percent`로 있고, "오늘"이라는 시간 경계는 원장 date로 이미 존재.

## 설계

### 1) '오늘 소비량(주간쿼터 대비)' 측정 — 스냅샷 델타
- 하루 첫 점검 때 **그날 시작 시점의 7d used_percent를 스냅샷**해 상태파일에 저장
  (`.yok3x/pace.json`: `{backend: {date, day_start_7d_pct}}`).
- `오늘_소비 = max(0, 현재_7d% − day_start_7d%)`. 이게 "오늘 주간쿼터를 몇 %p 태웠나".
- **정직한 한계**: 7d는 롤링 창이라 7일 전 사용분이 빠지면 델타가 실제보다 작아질 수 있음(음수는 0으로).
  근사 페이싱 신호로 충분하나, "정확한 당일 실사용"은 아님 → 리포트/툴팁에 명시.
- 라이브 7d가 없는 백엔드(gemini·실측실패)는 **원장 일일 예산**(기존 daily_tokens/calls)으로 폴백(명시적).

### 2) 한도·판정
- 설정 `guard.daily_pace`:
  - `enabled`(기본 false — opt-in), `pct_of_weekly`(예 0.2 = 하루 주간쿼터의 20%p),
  - `soft_frac`(예 0.8 → 캡의 80%에서 경고), `mode`("warn" | "pause").
- 판정: `오늘_소비 ≥ pct_of_weekly*100` → mode에 따라 `warn` 또는 `stop`.
  `≥ soft_frac*cap` → `warn`. 기존 5h/7d 절대 한도(soft/hard)와 **함께** 평가해 더 빡빡한 쪽 채택.

### 3) 일시정지 → 승인 재개(mode=pause)
- 캡 도달 → `check_backend`가 `stop`(metric="daily_pace") → 런 루프가 현행처럼 정지.
- **승인 재개**: `guard.daily_pace_override[backend] = "YYYY-MM-DD"`를 오늘로 세팅하면 그날은 페이싱 정지를
  건너뜀(절대 5h/7d 한도는 여전히 유효). CLI `yok3x pace-approve <backend>` + GUI 버튼으로 세팅.
- 자정 지나면 override 자동 무효(날짜 불일치).

### 4) GUI(설정 & 한도 탭) — §5.6 승인 필요
- 입력: 페이싱 on/off · `하루 상한(%)` 숫자 · mode 라디오(경고/정지-승인).
- 상태: 각 백엔드 "오늘 소비 X%p / 캡 Y%p" 미터. pause 상태면 **"오늘 재개 승인"** 버튼 노출.
- 데이터는 build_state로 내려주고 저장은 `/api/config`(기존 패턴). 표시방식은 리포트 승인 후 확정.

## 체크리스트(승인 후 진행)

- [ ] P2. `check_backend`에 daily_pace 판정 병합(warn/stop, override 반영). GuardVerdict metric="daily_pace".
- [ ] P3. config `guard.daily_pace{enabled,pct_of_weekly,soft_frac,mode}` + `daily_pace_override`.
- [ ] P4. 재개: `pace_approve(cfg, backend)` + CLI 서브커맨드 + `/api/config`(offline 토글과 동일 패턴).
- [ ] P5. GUI 설정탭 UI(입력·미터·재개 버튼) — **별도 승인**(§5.6).
- [ ] P6. 테스트(목): 델타 계산·경고/정지 전환·override 재개·자정 리셋·원장 폴백. 리포트·HISTORY·§5.5 등록.

## 결정 필요(사용자 확인)
1. **단위**: `pct_of_weekly`를 "주간쿼터의 %p"(권장, 직관적)로 갈지, 아니면 "주간한도÷7의 배수"로 갈지.
2. **범위**: 전역 1개 설정 vs 백엔드별 override(권장: 전역 + 선택적 백엔드별).
3. **mode 기본**: 안전하게 `warn` 기본 추천(pause는 opt-in).
4. GUI UI는 이 계획 승인 후 P5에서 별도 확정(§5.6).

## 원칙
- 의존성 0(상태파일은 stdlib json). 기존 5h/7d 절대 한도를 대체하지 않고 **덧붙이는 페이싱 층**.
- 명시적 열화·표시(§5.5), GUI는 승인 후만 수정(§5.6).
