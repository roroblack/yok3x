# v3.4.0 계획서 — Zed 에디터 연동 (ACP 에이전트로 yok3x 붙이기)

작성: 2026-07-11 · 결론 요약: **붙일 가치 있음. 방식은 "Zed 확장"이 아니라 "ACP 에이전트".
공식 Python SDK가 있어 마찰이 작다. 단, MVP 스파이크로 UX(지연) 먼저 검증 후 확대.**

---

## 1. 배경 — Zed의 외부 에이전트 연동은 ACP다

Zed(Rust·GPU 가속·오픈소스 에디터)는 외부 코딩 에이전트를 **ACP(Agent Client Protocol)**로
붙인다. 핵심 사실:

- LSP를 본뜬 **개방 표준**(Apache 라이선스). **JSON-RPC over stdio**. 에이전트는 에디터의
  서브프로세스로 실행된다.
- 역할 분담: **에디터(Zed)**가 스레드 UI·파일 접근·권한 프롬프트·터미널을 제공하고,
  **에이전트**가 런타임·인증·모델 선택·툴을 소유한다.
- **ACP 레지스트리**(Zed + JetBrains 공동, 2026-01)에 Claude Code·Codex CLI·Gemini CLI·
  OpenCode·Goose 등이 등재돼 에디터 안에서 나란히 돌아간다.
- **공식 Python SDK** 존재: `pip install agent-client-protocol`(3.10+). async 베이스클래스 +
  stdio JSON-RPC 배관 + 릴리스 추적 Pydantic 모델 → "커스텀 에이전트를 작게" 만들 수 있다.

즉 yok3x는 **Zed에 꽂히는 ACP 에이전트**가 되면 된다. yok3x가 파이썬이라 SDK 궁합이 좋다.

## 2. 핵심 질문 — Zed가 이미 claude/codex/gemini를 개별로 붙이는데 yok3x가 뭘 더 주나

이 질문에 답이 명확해서 **중복이 아니라 상위 레이어**라는 게 결론이다.

| yok3x 고유 가치 | Zed 개별 에이전트 |
|---|---|
| **producer-reviewer 교차검증**(한 모델 생성 → 다른 모델 채점) | 단일 에이전트 자기검증만 |
| **라이브 한도 가드**(claude_oauth·codex 실측 5h/7d) + **적응형 다운그레이드**(P1) | 없음 |
| **상황별 모델 라우팅**(v3.3 — 작업별 최적 모델 자동 선택) | 수동 선택 |
| pipeline·fanout-fanin·knot 공유기억·verify 게이트 | 없음 |

→ Zed 안에서 yok3x는 **"쿼터를 지키며 여러 모델을 교차검증으로 조율하는 오케스트레이터"**가
하나의 스레드로 나타난다. 개별 에이전트를 대체하는 게 아니라 그 위에 얹히는 레이어다.

**부수 효과(중요):** Zed에 붙으면 워커가 **실제 워크스페이스(열린 프로젝트)를 workdir로**
갖는다. → 최근 우리가 겪은 "workdir가 없어 프로듀서가 레포/브리프를 오인해 헤매는" 문제가
**자연 소멸**한다(Zed가 파일·권한을 ACP로 중개하므로). 이건 연동의 부가 이득이다.

## 3. 연동 방식 비교 — ACP가 정답

| 방식 | 적합도 | 이유 |
|---|---|---|
| **ACP 에이전트** | ✅ 채택 | 가장 깊고 네이티브. Agent Panel에 yok3x 스레드. 공식 파이썬 SDK로 마찰 최소 |
| MCP 컨텍스트 서버 | △ 보조 | MCP는 "다른 에이전트에 툴/컨텍스트 제공"용. yok3x는 *드라이버*라 부적합. 단 guard/coach를 Zed 네이티브 에이전트에 *툴*로 노출하는 보조 경로는 가능 |
| Zed 확장(WASM/Rust) | ✗ | API가 LSP/슬래시/테마로 제한 — 파이썬 오케스트레이션 실행 불가 |
| 태스크/터미널 | ✗(얕음) | `yok3x run`을 태스크로. 트리비얼, 스레드 통합 없음 |

## 4. ACP 구현 표면 (생각보다 작음)

**에이전트가 구현할 필수 메서드**: `initialize`(능력 협상) · `session/new` · `session/prompt`
(메인 루프) · `session/cancel`(알림). 선택: `authenticate` · `session/load` · `session/set_mode`.

**Zed(클라이언트)가 제공**: `session/update`(스트리밍 — 메시지 청크·툴콜·plan·모드) ·
`session/request_permission`. 선택: `fs/read_text_file`·`fs/write_text_file`·`terminal/*`.

**yok3x 매핑(깔끔함):**
- `session/prompt` → `orchestrator.run_producer_reviewer`/`run_pipeline` 한 번.
- 각 step(프로듀서 생성 · 리뷰어 SCORE · 가드/coach 상태) → `session/update`로 스트리밍
  (step = plan 엔트리 / 툴콜 블록으로 표현).
- 기존 **승인 게이트**(`_gate`) → `session/request_permission`으로 위임(Zed UI가 y/n).
- **가드 stop / coach 라우팅** → 상태 메시지 청크로 노출("claude 92% → codex 위임" 등).
- 파일 편집: workdir가 실제 프로젝트이므로 워커가 편집 → Zed의 fs/permission으로 중개.

## 5. 규모 산정

- **의존성 0 원칙 유지**: ACP SDK는 dep이므로 **어댑터만 optional extra**로 격리
  (`pip install yok3x[zed]`). 코어(claude_oauth·guard·orchestrator)는 dep-0 그대로.
- **MVP**: `yok3x acp` 명령 = ACP 에이전트 진입점. `initialize`+`session/new`+`session/prompt`
  1개(= producer-reviewer 1회), step을 `session/update`로 스트리밍, 권한 게이트 연결.
  **약 1~2일.**
- **확장**: plan/툴콜 블록 정교화, `session/cancel`, 멀티턴 세션(`session/load`), 파일 diff
  표현, v3.3 라우팅 결합. 반복적.

## 6. 솔직한 리스크와 대응

| 리스크 | 대응 |
|---|---|
| **중복 우려** — "그냥 Claude만 쓰고 싶다"는 사용자에겐 불필요 | 교차검증+쿼터가드로 가치 증명. "빠른 편집"이 아니라 "철저·검증" 작업용으로 포지셔닝 |
| **지연/무게** — Zed→yok3x→CLI 3개 깊은 스택. producer-reviewer 한 턴이 수 분(codex 느림) → 에디터에서 굼떠 보임 | MVP UX 검증으로 go/no-go. 스트리밍으로 진행 가시화. 빠른 단일 에이전트와 보완 관계로 사용 |
| 의존성 0 훼손 | ACP 어댑터를 optional extra로 격리 |
| ACP 프로토콜 미성숙(2025~26) | 릴리스 추적 Pydantic SDK 사용, 어댑터를 얇게 유지 |

## 7. 권고 — 단계

1. **P0: MVP 스파이크(1~2일)** — `yok3x acp` 어댑터(optional extra). producer-reviewer 1턴을
   Zed Agent Panel에 스트리밍. 커스텀 에이전트로 등록해 실제로 붙여본다.
2. **P0 게이트: UX 검증** — 다중에이전트 지연이 에디터에서 견딜 만한가? 여기서 확대 여부 결정.
3. **P1: 정교화** — 권한/plan/취소/파일 diff, 멀티턴 세션.
4. **P2: 라우팅 결합** — v3.3 상황별 프로파일을 붙여 "에디터 안에서 작업별 최적 모델 자동
   선택 + 교차검증 + 쿼터가드"라는 완성형 포지션.

**한 줄 결론:** yok3x의 오케스트레이션·쿼터가드는 Zed의 개별 에이전트가 못 주는 상위 가치라
연동은 타당하다. ACP + 공식 파이썬 SDK로 진입 장벽도 낮다. 다만 **인터랙티브 지연**이 유일한
실질 리스크이므로, 큰 투자 전에 MVP로 UX부터 확인한다.
