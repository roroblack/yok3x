# v3.8.0 · 계획서 — GUI 작업(task) 관리 CRUD · 2026-07-14

## 목표
GUI에서 작업을 **이름 붙여 저장 / 목록에서 열기·재실행 / 편집 / 삭제**. 현재는 인라인 즉시실행만
있고 명명된 작업 CRUD가 없다. 방금 만든 **라벨(작업별 콘솔)**과 연결.

## 현재 구조(근거)
- 작업 파일 = 프로젝트 루트 `task-*.json`(pattern/task/producer/reviewer/label/max_rounds…).
- `_list_tasks`가 나열 → `state.tasks`. `/api/run`이 등록 task 파일 또는 인라인 spec(`.yok3x/console/` 임시) 실행.
- `SAMPLE_TASKS`가 초기 3종 생성. 즉 **저장·실행 기반은 이미 있고, CRUD UI/API만 없음**.

## codex 논의 결과(반영)
1. 저장은 루트 `task-<slug>.json` 권장(기존 호환), `.yok3x/console/` 임시는 CRUD 목록에서 제외.
2. '열기'=JSON 검증 후 pattern/task/producer/reviewer/label을 폼에 로드, 원본 파일명을 편집 상태로 유지.
3. 삭제는 확인 후 허용된 `task-*.json`만. 이름 중복=덮어쓰기 확인 또는 새 이름.
4. 새 CRUD API는 `_list_tasks`·`/api/run` 재사용, 표준 파일/JSON만(의존성0), 저장은 **임시파일→원자적 교체**.
5. 엣지: 빈/금지 이름, **경로순회(`../`)**, 잘못된 JSON·필수필드 누락, 외부 변경·삭제, 실행 중 삭제.

## 설계

### 1) 백엔드 API(guiserver, 의존성0)
- `GET /api/task?name=task-x.json` → 파일 내용(JSON) 반환. 이름 검증(아래).
- `POST /api/task` `{name, spec}` → 저장/편집. 원자적(temp→replace). 검증 통과 시만.
- `POST /api/task/delete` `{name}` → 삭제(허용 파일만). (DELETE 메서드 대신 POST로 통일)
- 목록은 기존 `state.tasks`(_list_tasks) 재사용.
- **이름 검증(핵심 보안)**: `task-<slug>.json` 형식만 허용. slug=`[a-z0-9-]`, `..`/경로구분자 금지,
  `cfg.paths.root` 밖 경로 거부(resolve 후 root 하위 확인). spec은 pattern·task 필수 검증(기존 /api/run 규칙 재사용).

### 2) GUI(§5.6 — 이 계획 승인 후)
- 콘솔에 **"저장된 작업"** 영역: 드롭다운/리스트(이름) + [열기] [삭제] 버튼. (작업별 콘솔 라벨과 나란히)
- 런 입력 폼에 **[작업 저장]** 버튼 + 이름 입력 → 현재 폼(pattern/task/워커/라벨)을 `task-<slug>.json`로 저장.
- **열기** → 폼에 로드(편집 모드, 원본명 표시). **저장** 시 같은 이름=덮어쓰기 확인, 새 이름=신규.
- **삭제** → 확인 다이얼로그 후 제거. 실행 중인 작업은 삭제 경고.

### 3) 라벨 연동
- 저장 시 spec.label 기본=작업 이름(slug). 실행하면 콘솔 '작업별 보기'에서 그 라벨로 묶임 → 저장~실행~조회 일관.
