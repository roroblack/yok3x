# v3.2.0 계획서 — 한도 인근 적응형 열화(모델 다운그레이드 · 백엔드 전환)

> **구현 상태(2026-07-11)**: **P1(모델 다운그레이드) 완료** — `degrade_plan` + `model_arg`
> 주입, opt-in, 리뷰어 제외, 명시 로깅. P2(백엔드 failover)·P3(GUI) 미착수.

작성: 2026-07-10 · 대상: 요금 가드가 `stop`으로 루프를 죽이기 **전에**, 사용률이 높아지면
모델을 낮추거나 여유 있는 도구로 갈아타 **작업을 계속 이어가는** 기능.

---

## 1. 왜 (동기)

지금 가드는 2단계다: `soft`(경고) → `hard`(정지). hard에 닿으면 `RunAborted`로 루프가
멈춘다. 하지만 현실에선 "한도가 꽉 차간다"와 "완전히 막혔다" 사이에 **쓸 수 있는 중간
대응**이 있다:

- **모델 다운그레이드** — 같은 도구 안에서 더 가벼운 모델로 낮춰(예: Opus→Haiku, codex
  reasoning high→low) 남은 한도로 더 오래 버틴다.
- **백엔드 전환(failover)** — 그 도구를 잠시 접고, 여유 있는 다른 도구로 해당 워커(또는
  오케스트레이터 메인 모델)를 임시 재배정한다. coach가 이미 "지금 gemini로 돌려라"라고
  **말**은 하는데, 그걸 **자동 실행**으로 승격하는 것.

즉 가드를 `ok → warn → 다운그레이드 → 전환 → stop`의 5단계로 확장하는 제안이다.

## 2. 현재 상태 (재사용 가능한 뼈대)

| 요소 | 현재 | 재사용 |
|---|---|---|
| 사용률 실측 | `limits.probe` → 5h/7d `used_percent`·리셋 (claude/codex 실측, gemini 원장) | 판단 근거 그대로 사용 |
| 가드 판정 | `usage._level(g, ratio)` → ok/warn/stop | 중간 단계 2개만 끼워넣으면 됨 |
| 여유 도구 계산 | `coach_messages`가 `freest`/`alt` 산출 | 전환 대상 선정 로직 재사용 |
| 워커→백엔드 | `cfg.worker(name)["backend"]` → `backends.json` command 템플릿 | 런타임 오버라이드 지점 |
| 실행 진입 | `orchestrator.call_worker` 시작부(가드 체크 직후) | 열화 훅을 넣을 단일 지점 |

핵심: **하드코딩 없이 대부분 이미 있는 것들의 조합**이다. 새로 필요한 건 (a) 모델 티어
설정, (b) command에 모델 인자 주입, (c) call_worker의 런타임 오버라이드다.

## 3. 설계

### 3.1 5단계 가드
```
r = 현재 사용률(가장 빡빡한 창 기준, limits.probe)
r < soft(0.8)            → 정상
soft ≤ r < downgrade     → 경고 (현행)
downgrade ≤ r < failover → 모델 다운그레이드 (같은 백엔드, 가벼운 모델)
failover ≤ r < hard      → 백엔드 전환 (여유 도구로 워커 재배정)
r ≥ hard(1.0)            → 정지 (현행)
```

### 3.2 설정 스키마 (yok3x.json)
```jsonc
"guard": {
  "soft_ratio": 0.8, "hard_ratio": 1.0,
  "degrade": {
    "enabled": true,
    "downgrade_ratio": 0.9,     // 이 사용률↑ → 모델 다운그레이드
    "failover_ratio": 0.97,     // 이 사용률↑ → 다른 백엔드로 전환
    "recover_ratio": 0.75,      // 이 아래로 내려가야 원복(히스테리시스, 왕복 방지)
    "roles_no_downgrade": ["codex-critic"], // 리뷰어는 품질 게이트라 다운그레이드 제외
    "max_switches_per_run": 3               // 한 런 내 전환 상한(스래싱 방지)
  }
},
"limits": {
  "claude": { ..., "models": { "full": "", "lite": "claude-haiku-4-5-20251001" } },
  "codex":  { ..., "models": { "full": "", "lite": "gpt-5-codex-low" } }
}
```
- `full: ""` = CLI 기본 모델(현행 그대로). `lite` = 다운그레이드 타깃. 모델 ID는 드리프트가
  잦으니 **코드 하드코딩이 아니라 설정으로** 둔다.

### 3.3 command에 모델 주입
`backends.json` command에 선택적 `{model}` 자리를 두거나, 백엔드별 `model_arg` 템플릿을
둔다:
```jsonc
"claude": { "command": ["claude","-p","{prompt}","--output-format","json"],
            "model_arg": ["--model","{model}"] }   // 다운그레이드 시에만 append
```
`_run_cli`가 선택된 모델이 있으면 `model_arg`를 치환·추가한다(없으면 현행과 100% 동일).

### 3.4 런타임 훅 (`call_worker`)
가드 체크 직후, 실행 전에:
```
verdict = usage.check_backend(cfg, backend)      # 이미 있음
plan = degrade_plan(cfg, worker, verdict, run_state)   # 신설(순수 함수)
  → ("normal", None) | ("downgrade", model_id) | ("failover", other_backend) | ("stop", reason)
```
- `downgrade` → 이번 호출에 `model_arg` 주입.
- `failover` → `backend`를 `verdict`/coach의 여유 도구로 교체(워커 role 프롬프트는 유지).
- 모든 열화는 **StepLog·run.log·status에 명시 기록**하고 coach/GUI에 배지로 노출
  (RULE §5.5: 조용한 동작 변경 금지). 예: `[degrade] claude 92% → opus 대신 haiku`,
  `[failover] claude 98% → gemini로 이번 라운드 위임`.
- 히스테리시스: `recover_ratio` 아래로 내려가기 전엔 원복 안 함. `max_switches_per_run`로
  왕복 상한.

### 3.5 GUI/coach 노출
- 대시보드 게이지에 이미 **임계선(soft/hard)**이 있으니, `downgrade_ratio`·`failover_ratio`
  선도 추가(색: 주황·자주). 현재 활성 열화는 워커 카드에 배지("haiku로 다운" / "→gemini").
- coach 문구를 "권고"에서 "적용됨"으로: `[전환·실측] claude 98% → 이번 라운드 gemini 위임`.

## 4. 내 평가 (권고)

**타당하고 가치 있다. 단, 두 가지 원칙을 지켜야 한다.**

1. **투명성 우선.** 자동으로 모델을 바꾸면 결과물 품질/성격이 달라진다. 조용히 하면
   "왜 갑자기 답이 부실하지?"가 된다. **모든 열화를 로그·상태·GUI에 명시**하는 걸
   기능의 1급 요구사항으로 둔다(이 프로젝트의 §5.5·§7 정신과 정확히 일치).
2. **역할 인식.** producer는 다운그레이드해도 되지만 **reviewer(codex-critic)를 낮추면
   품질 게이트 자체가 약해진다** — producer-reviewer의 존재 이유가 무너진다. 그래서
   `roles_no_downgrade`로 리뷰어는 기본 제외. 리뷰어가 한도면 **다운그레이드보다
   백엔드 전환**(다른 실측-여유 도구)이 옳다.

그리고 **스래싱 방지**(히스테리시스 + 런당 상한)와 **모델 ID 설정화**(하드코딩 금지)는
필수. 이 둘이 없으면 "일단 돌아가게"식 폴백이 되어 §5.5 위반이 된다.

결론: **하되, opt-in + 투명 + 역할 인식**으로. 처음부터 자동 전면 적용은 위험하니
단계적으로 켠다.

## 5. 단계별 실행안

| 단계 | 범위 | 산출 |
|---|---|---|
| **P1** | 모델 다운그레이드(producer 한정, opt-in). 설정 스키마 + `model_arg` 주입 + `degrade_plan` 순수함수 + 명시 로깅 | 가장 안전한 최소 가치. 리뷰어 불변 |
| **P2** | 백엔드 failover(여유 도구 재배정) + 히스테리시스 + 런당 상한 | coach의 "권고"를 "자동"으로 승격 |
| **P3** | GUI 노출(열화 임계선·활성 배지) + coach 문구 전환 + 회귀 테스트 스위트 | 사용자가 무슨 일이 일어나는지 한눈에 |

### 테스트(각 단계)
- `degrade_plan`은 순수 함수라 **결정적 단위 테스트** 쉬움: (ratio, role, run_state) →
  기대 액션. 히스테리시스·상한·리뷰어 제외 케이스 잠금.
- mock 백엔드로 "downgrade 시 `--model` 주입됨", "failover 시 backend 교체됨",
  "리뷰어는 다운그레이드 안 됨"을 end-to-end 검증.

## 6. 리스크와 대응

| 리스크 | 대응 |
|---|---|
| 자동 전환이 사용자 의도 밖 도구/계정에 비용 발생 | opt-in 기본 off, failover 대상은 실측-여유 도구로 한정, 로그 명시 |
| 모델 ID 드리프트(이름 변경) | 코드 하드코딩 금지 → `limits.*.models` 설정값 사용 |
| 다운그레이드로 품질 저하 인지 못 함 | 상태/GUI/coach 명시 배지 필수(1급 요구사항) |
| 왕복 스래싱 | `recover_ratio` 히스테리시스 + `max_switches_per_run` |
| 리뷰어 약화로 게이트 무력화 | `roles_no_downgrade` 기본값에 리뷰어 포함 |

## 7. 예상 규모
P1 ≈ 반나절(설정 + 주입 + 순수함수 + 테스트). P2 ≈ 반나절. P3 ≈ GUI 손보기.
새 의존성 0(현행 유지). 기존 가드/coach/limits와 충돌 없음(중간 단계 삽입일 뿐).
