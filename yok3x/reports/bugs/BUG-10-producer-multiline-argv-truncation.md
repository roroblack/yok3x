# BUG-10 · 프로듀서 병목 — 멀티라인 argv 잘림 (치명, 다세션 미스터리)

- **시점**: 2026-07-10~11 · **심각도**: 치명 · **커밋**: `43ec4db` · **상태**: 수정됨 ✅

## 증상
producer-reviewer에서 프로듀서(claude)가 코드를 내지 않고 **"작업 요청이 없다 / 대기하겠다"**
또는 **"[작업] 태그만 있고 실제 작업 내용이 비어 있다"**고 응답. codex 리뷰어는 그걸 보고
정직하게 SCORE 0~5. 여러 세션에 걸쳐 "확률적"으로 보였다.

## 오진(정직 기록)
근본 원인을 찾기까지 여러 헛다리를 짚었다:
- 권한 모드(bypassPermissions) → 오히려 에이전트 모드로 파일 뒤짐(악화).
- 격리 cwd(BUG-08) → 레포 오염은 줄었으나 증상 잔존.
- role의 "파일 편집·경로 보고" 언어, anti-hallucination, brief/context 플레이스홀더 → 이분탐색했으나
  같은 프롬프트가 **직접 호출에선 코드 반환**해 프롬프트 텍스트 문제 아님으로 판명.
- "확률적 모델 행동"으로 잠정 결론(틀림).

## 근본 원인 (결정적)
**Windows에서 claude/codex/gemini는 npm `.cmd` 심**이고, 멀티라인 문자열을 subprocess **argv**로
넘기면 **cmd.exe가 첫 줄바꿈에서 인자를 잘라버린다**. 프롬프트가 `[작업]\n실제내용…`이면 워커는
**첫 줄 `[작업]`만 받고** 본문을 못 받아 "내용이 비어 있다"고 응답. 한 줄 프롬프트만 우연히
동작해서 "확률적"으로 보였을 뿐, **결정적 버그**였다.

## 진단 (결정적 대조)
같은 멀티라인 프롬프트를 argv vs stdin으로 전달해 대조:
```
A) argv 멀티라인: FAIL ("작업 내용이 비어 있어서…")
B) stdin:        CODE ("```python\ndef is_palindrome…")
```

## 수정
`_run_cli`: 커맨드 argv에 `{prompt}`가 없으면 프롬프트를 **stdin(`input=`)** 으로 전달. 세 CLI
커맨드에서 `{prompt}` 제거(stdin 모드). `input=`은 프롬프트 후 즉시 EOF라 데드락도 방지.
겸사 코드생성 프롬프트를 미니멀화 + role의 파일-편집 언어 제거(정합성).

## 검증
run_backend 멀티라인 성공률 **0/4 → 4/4 CODE**. 실제 producer-reviewer **SCORE 7.0 통과**(done).
단위 테스트 `test_cli_backend_passes_multiline_prompt_via_stdin`(argv엔 없고 `input=`에 온전 전달).

## 교훈
- Windows에서 **CLI에 멀티라인 텍스트를 argv로 넘기지 마라** — `.cmd` 심/cmd.exe가 자른다. **stdin**으로.
- "확률적"으로 보이면 의심하라 — 대개 **입력의 어떤 성질(여기선 줄바꿈 유무)에 따른 결정적** 동작이다.
- 오진을 정직하게 기록한다: 권한·cwd·프롬프트는 모두 진짜 원인이 아니었다.
