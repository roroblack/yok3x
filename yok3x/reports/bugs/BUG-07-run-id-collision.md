# BUG-07 · run_id 초 단위 충돌 → 동시 런 손상

- **시점**: 2026-07-11 · **심각도**: 중간 · **커밋**: `5d00b00` · **상태**: 수정됨 ✅

## 증상
검증용으로 런을 겹쳐 돌리던 중, 한 런 디렉터리에 `step_01`이 없고 `step_04`만 남거나
`score=0.0` 같은 **손상된 결과**가 나왔다. 어떤 코드 버전이 돌았는지도 뒤섞였다.

## 근본 원인
`Orchestrator.run_id = datetime.now().strftime("run_%Y%m%d_%H%M%S")` — **초 단위** 해상도.
같은 초에 시작한 동시 런이 **같은 `run_dir`를 공유**해 서로의 step 파일·status.json을 덮어썼다.
GUI는 큐로 직렬화하지만 CLI 동시 실행·GUI+CLI 병행에서 발생 가능.

## 진단
겹친 백그라운드 런들의 디스크 상태를 대조 → 동일 타임스탬프 run_dir 충돌로 특정.

## 수정
`strftime("run_%Y%m%d_%H%M%S_%f")` — **마이크로초** 포함으로 유니크화.

## 검증
단위 테스트 `test_run_id_includes_microseconds`(형식 `run_\d{8}_\d{6}_\d{6}` 정규식).

## 교훈
런/작업 ID는 **동시성 하에 유일**해야 한다. 초 단위 타임스탬프는 병렬에서 충돌한다 —
마이크로초/카운터/uuid로 유일성 확보.
