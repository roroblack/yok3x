# 분석 — claude OAuth 토큰 자체 갱신: 차단 위험·설계 · 2026-07-14

## 질문
yok3x가 claude 토큰을 자체 갱신하면 (1) 클로드가 차단할 수 있나? (2) 위험 있으면 갱신 주기에
랜덤성을 넣는 게 좋나?

## 답: 토큰 갱신 자체는 차단 대상이 아니다
- OAuth **refresh_token으로 access_token을 재발급**받는 건 **표준·정상 동작**이다. Claude Code(공식 CLI)가
  바로 이렇게 자동 갱신한다. 자기 토큰을 표준 엔드포인트로 갱신하는 것 = 위반 아님 → 그것만으로 밴 안 됨.
- 현실적 위험은 '차단'이 아니라 **레이트리밋(429)**: 갱신을 **너무 자주**(초 단위) 때리면 엔드포인트
  한도에 걸린다. 정상 주기(만료 임박 시 ~1시간에 1번)면 문제 없음.

## '랜덤성으로 회피' 프레이밍은 부적절 — 대신 '착한 클라이언트'
- 탐지를 피하려 지터를 넣는 접근은 **회피(evasion) 프레이밍**이라 지향하지 않는다. 그리고 **필요도 없다.**
- 올바른 설계는 저절로 정상 주기가 된다:
  1. **만료 임박에만 갱신**(예: 남은 시간 < 5분). 고정 타이머로 자주 때리지 말 것 → 자연히 ~1시간에 1번,
     공식 클라이언트와 구분 불가. 회피 장치 불필요.
  2. **실패 시 지수 백오프 + 소량 지터** — 이건 회피가 아니라 **재시도 폭주(thundering herd) 방지**의
     표준 엔지니어링. 429/오류 시에만 적용.
- 즉 "차단 위험 → 랜덤성"이 아니라 "**만료 임박 갱신 + 실패 시 백오프**"가 정답. 결과적으로 안전·정상.

## 설계 제안 (구현 후보)
- `limits.py`: probe 전, access_token 만료까지 `< refresh_margin_sec`(기본 300)면 refresh_token으로 재발급.
  - Anthropic 토큰 엔드포인트에 POST(표준 OAuth refresh). 성공 시 `.credentials.json` 원자적 갱신.
  - **호출 억제**: `min_refresh_interval_sec`(기본 60)로 과다 갱신 차단 + 실패 시 지수 백오프(+지터).
  - 실패해도 기존 폴백(stale/추정/원장)이 있으니 무중단(fail-safe).
- 자격 처리: 사용자 **자기 토큰**을 표준 플로우로 갱신(폼 입력 아님, Claude Code와 동일 방식). 로컬 전용.
- 이점: claude 실측(5h/7d)이 토큰 만료로 끊기던 반복 불편 해소 → 하루 페이싱 등도 실측 유지.

## 판단
- **차단 위험 낮음**(정상 동작). 회피용 랜덤성 불필요·부적절.
- **near-expiry 갱신 + 실패 백오프**로 구현하면 안전. 계획서로 진행 가능(우선순위: 중).
