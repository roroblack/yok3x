# v3.3.0 계획서 — 상황별 최적 모델 프로파일(situational model profiles)

> **구현 상태(2026-07-11)**: **S1·S2·S3 완료**. S1=정적 프로파일. S2=가용성·한도 필터
> (`backend_available`+benchmarks 폴백). S3=`best` argmax 자동유도(`_derive` — benchmarks
> 최고점 채택) + GUI 프로파일 선택기·"왜 이 모델" 미리보기. 새 backend(GLM 등)는 설정으로
> 추가 가능하나 실 CLI 없어 미검증(claude/codex/gemini만 실동작).

작성: 2026-07-11 · 목표: "이 작업엔 어떤 모델이 최선인가"를 벤치마크 근거로 **자동 선택**.
사용자는 목표(mode)만 고르면, yok3x가 작업 종류마다 최적 backend·model을 라우팅한다.

---

## 1. 왜 (동기)

벤치마크 현실(사용자 제공 스냅샷 요약): **모든 코딩 작업을 지배하는 단일 모델은 없다.**

| 상황 | 강자 |
|---|---|
| 정확한 레포 패치(SWE-Bench Pro) | Fable 5 > Opus 4.8 > GPT-5.6 |
| 터미널·빌드·테스트 반복(Terminal-Bench) | GPT-5.6 Sol > GPT-5.5 > Fable 5 |
| 장시간 코드베이스(DeepSWE) | GPT-5.6 > Fable 5 |
| 깊은 구조 분석(FrontierSWE) | Opus 4.8 > GLM-5.2 |
| MCP·도구 연결(MCP Atlas) | Gemini 3.5 |
| 가성비/속도 | Sonnet 5 · Gemini Flash · GLM-5.2 |

그런데 yok3x의 `routing`은 지금 **작업→backend(도구)** 만 고르고 **모델**은 CLI 기본값을
쓴다. 즉 "리뷰는 정확도가 중요하니 Fable 5로, 터미널 반복은 GPT-5.6 Sol로" 같은
상황 최적화를 못 한다. 이걸 **모델 단위**로 끌어올리자는 제안이다.

## 2. 이미 있는 배관 (재사용)

**P1(v3.2.0)에서 per-call 모델 주입을 이미 만들었다** — 이 기능의 절반은 완성돼 있다.

| 요소 | 상태 |
|---|---|
| `backends.*.model_arg` + `run_backend(model=)` | ✅ 완료(P1) — 호출마다 `--model` 주입 |
| `limits.*.models` | ✅ 존재(full/lite) — 여기에 카탈로그 확장 |
| `routing`(작업→backend) | ✅ 존재 — 모델까지 담도록 확장 |
| `usage.check_backend`(실측 사용률) | ✅ — 한도 여유로 라우팅 필터 |
| CLI 설치·인증 감지 | 신설 필요(shutil.which + 경량 probe) |

즉 **새로 짤 건 (a) 모델 카탈로그·프로파일 스키마, (b) 상황→모델 해석기, (c) 가용성/한도
필터**뿐이다. 실행 배관은 P1로 이미 깔렸다.

## 3. 설계

### 3.1 3층 구조
```
mode(목표)  ×  task-kind(상황)  →  model(논리)  →  (backend, model_id)
   best         precise-patch      Fable 5          claude, claude-fable-5
   balanced     terminal-loop      GPT-5.6 Sol      codex,  gpt-5.6-sol
   cost         review             Sonnet 5         claude, claude-sonnet-5
   speed        tool-mcp           Gemini 3.5       gemini, gemini-3.5-flash
```

### 3.2 모델 카탈로그 (설정, 사용자 갱신 가능)
```jsonc
"models_catalog": {
  "fable-5":   { "backend": "claude", "model": "claude-fable-5",         "cost": 5, "speed": 2 },
  "opus-4.8":  { "backend": "claude", "model": "claude-opus-4-8",        "cost": 4, "speed": 2 },
  "sonnet-5":  { "backend": "claude", "model": "claude-sonnet-5",        "cost": 2, "speed": 4 },
  "haiku-4.5": { "backend": "claude", "model": "claude-haiku-4-5-20251001","cost": 1, "speed": 5 },
  "gpt-5.6":   { "backend": "codex",  "model": "gpt-5.6-sol",            "cost": 3, "speed": 4 },
  "gemini-3.5":{ "backend": "gemini", "model": "gemini-3.5-flash",       "cost": 1, "speed": 5 },
  "glm-5.2":   { "backend": "glm",    "model": "glm-5.2",                "cost": 1, "speed": 3 }
}
```
- claude 계열 model_id는 실제 값(claude-fable-5 / claude-opus-4-8 / claude-sonnet-5 /
  claude-haiku-4-5-20251001). codex/gemini/glm는 사용자가 자기 환경 값으로 채운다.
- **GLM 등 새 도구는 backend 어댑터 추가 필요**(현재 claude/codex/gemini뿐).

### 3.3 벤치마크 데이터 (설정 파일 = 근거, 하드코딩 금지)
```jsonc
"benchmarks": {          // 상황별 점수 — 사용자가 최신값으로 갱신
  "precise-patch": { "fable-5": 80.0, "opus-4.8": 69.2, "gpt-5.6": 64.6, "sonnet-5": 63.2 },
  "terminal-loop": { "gpt-5.6": 88.8, "fable-5": 83.1, "glm-5.2": 81.0, "sonnet-5": 80.4 },
  "long-horizon":  { "gpt-5.6": 72.7, "fable-5": 69.7, "opus-4.8": 59.0 },
  "deep-analysis": { "opus-4.8": 75.1, "glm-5.2": 74.4 },
  "tool-mcp":      { "gemini-3.5": 83.6, "opus-4.8": 77.8, "glm-5.2": 76.8 }
}
```
- 벤치마크는 하네스·effort·문제셋에 따라 달라진다(사용자도 지적). 그래서 **점수를 코드에
  박지 않고 설정으로** 둬서, 각자 신뢰하는 출처로 갱신·재정렬하게 한다.

### 3.4 프로파일(mode) = 상황→모델 매핑
```jsonc
"profiles": {
  "best":     { "precise-patch":"fable-5","terminal-loop":"gpt-5.6","review":"fable-5",
                "deep-analysis":"opus-4.8","tool-mcp":"gemini-3.5","long-horizon":"gpt-5.6" },
  "balanced": { "precise-patch":"opus-4.8","terminal-loop":"gpt-5.6","review":"opus-4.8",
                "tool-mcp":"gemini-3.5","long-horizon":"gpt-5.6" },
  "cost":     { "*":"sonnet-5", "tool-mcp":"gemini-3.5" },
  "speed":    { "*":"gemini-3.5", "precise-patch":"sonnet-5" },
  "self-host":{ "*":"glm-5.2" }
},
"active_profile": "balanced"
```
- `"*"`는 기본값, 특정 상황만 오버라이드. `best`는 점수표에서 상황별 1위를 자동 채택하는
  **derive 모드**도 가능(benchmarks에서 argmax).

### 3.5 상황(task-kind) 결정
워커 호출 시 이미 있는 `task_kind`(build/refactor/critic/review/test/design_review)와
worker role을 상황으로 매핑:
- `critic`/`review` → precise-patch 성격(정확도) → review 슬롯
- `build`+workdir(장시간) → long-horizon
- `build`(단발) → terminal-loop 또는 precise-patch
- `design_review` → deep-analysis / tool-mcp
(간단한 매핑 테이블 + 사용자 오버라이드)

### 3.6 해석기 (순수 함수, P1의 model_arg로 실행)
```
resolve_model(cfg, worker, task_kind)
  → 프로파일[active][situation] → 카탈로그 → (backend, model_id)
  → 가용성 필터: backend CLI 설치·인증? 한도 여유(check_backend != stop)?
     아니면 그 상황 점수표에서 '가용한 다음 순위'로 폴백
  → call_worker가 backend override + model 주입(P1 배관 그대로)
```
모든 선택은 **명시 로깅·GUI 표시**: `[route] review → fable-5(claude) (best 프로파일, 정확도 1위)`.

## 4. 내 평가 (권고)

**타당하고, P1 덕분에 착수 비용이 낮다. 네 가지 원칙을 지켜라.**

1. **벤치마크는 데이터로, 코드로 박지 마라.** 점수는 하네스·시점마다 흔들리고(사용자도
   강조) 새 모델이 계속 나온다. 설정 파일로 두고 argmax로 프로파일을 유도하면, 갱신이
   설정 한 줄로 끝난다. 이게 §5.5(하드코딩 최소화)·§7(정직성)과 맞다.
2. **가용성·한도 인식 필수.** "이론상 최고 모델"이 이 컴퓨터에 설치·인증 안 됐거나 한도가
   꽉 찼으면 무용지물이다. 반드시 **설치·인증·여유를 확인하고, 안 되면 그 상황의 다음
   순위로 폴백**한다(적응형 열화 P1/P2와 자연스럽게 합쳐진다: 프로파일=full 티어 선택,
   열화=한도 근처에서 lite로 낮춤).
3. **투명성.** 어떤 모델이 왜 선택됐는지(프로파일·상황·근거 점수) 로그·GUI에 노출. 조용히
   모델이 바뀌면 결과 성격이 달라져 혼란을 준다.
4. **사용자 오버라이드가 항상 우선.** 프로파일은 '기본 추천'이고, 태스크 파일의 명시
   `producer`/`reviewer`/`model`이 있으면 그게 이긴다.

**주의점**: 사용자가 붙인 점수표엔 조건이 뒤섞인 값(†=타사 환경, Ultra=멀티에이전트)이
섞여 있다. 프로파일 유도에 쓸 땐 **같은 하네스·effort 기준으로만 비교**하도록 벤치마크
항목에 `condition` 태그를 달아 섞이지 않게 한다. 안 그러면 "최고 모델"이 착시가 된다.

결론: **하자. 단 (a) 벤치마크=설정데이터, (b) 가용성·한도 필터, (c) 투명 로깅,
(d) 사용자 우선.** P1의 model_arg가 이미 있어 P1단계는 며칠이 아니라 반나절 규모.

## 5. 단계별 실행안

| 단계 | 범위 | 산출 |
|---|---|---|
| **S1** | 정적 프로파일: models_catalog + profiles + `resolve_model` 순수함수 + call_worker에서 backend·model 오버라이드(P1 배관 재사용) + `yok3x profile <mode>` CLI + 명시 로깅 | 목표만 고르면 상황별 모델 자동 선택 |
| **S2** | 가용성·한도 필터: CLI 설치·인증 감지 + check_backend 여유 확인 → 안 되면 점수표 다음 순위 폴백. 적응형 열화(P1/P2)와 통합 | "이론상 최고"가 아니라 "지금 가능한 최고" |
| **S3** | 벤치마크 데이터로 `best` 프로파일 자동 유도(argmax) + GUI 프로파일 선택기·"왜 이 모델" 표시 + 새 backend 어댑터(GLM 등) | 데이터 갱신만으로 최신화, 한눈에 근거 |

### 테스트
- `resolve_model`은 순수 함수 → 결정적 단위 테스트: (mode, task_kind, 가용성) → 기대 (backend, model). 폴백·오버라이드·derive 케이스 잠금.
- mock 백엔드로 "best 프로파일에서 review→fable 모델 주입됨", "backend 미설치 시 다음 순위로 폴백" end-to-end.

## 6. 리스크와 대응
| 리스크 | 대응 |
|---|---|
| 벤치마크 값 드리프트·조건 혼재 | 설정 데이터 + `condition` 태그로 동일 조건만 비교 |
| 이론상 최고 모델이 미설치/한도초과 | 가용성·한도 필터 + 다음 순위 폴백(S2) |
| 모델 자동 전환으로 결과 성격 변동 인지 못 함 | 선택 근거 명시 로깅·GUI(투명성) |
| 새 backend(GLM 등) 필요 | S3에서 어댑터 추가, 그전엔 claude/codex/gemini만 |
| 사용자 의도 무시 | 태스크 명시 지정이 프로파일보다 우선 |

## 7. 예상 규모
S1 ≈ 반나절(P1 배관 재사용). S2 ≈ 반나절. S3 ≈ GUI + 어댑터. 새 의존성 0.
기존 routing/limits/degrade와 충돌 없음(같은 축의 상위 레이어).

## 8. 이 프로젝트에 바로 붙는 기본 프로파일(참고)
현재 설치된 것(claude=Fable/Opus/Sonnet/Haiku, codex, gemini) 기준 `balanced` 예시:
- review/critic(정확도) → **Opus 4.8**(claude) · best면 **Fable 5**
- build 단발/터미널 → **GPT-5.6 Sol**(codex)
- 장시간 workdir → **GPT-5.6 Sol**(codex) 또는 **Fable 5**(claude)
- design_review/도구 → **Gemini**(gemini)
- 비용 절감/다운그레이드 → **Sonnet 5 → Haiku 4.5**(claude, 적응형 열화와 연동)
