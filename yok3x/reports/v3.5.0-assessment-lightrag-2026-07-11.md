# v3.5.0 평가 — LightRAG yok3x 적용 검토

작성: 2026-07-11 · 출처: LightRAG: Simple and Fast Retrieval-Augmented Generation
(arXiv:2410.05779, HKUDS/LightRAG) · **결론: 프레임워크 통째 도입은 부적합(의존성 0 원칙 위배).
단, 핵심 아이디어(그래프 탐색·이중레벨 검색)를 `knot`에 경량 이식하는 건 가치 있음.**

---

## 1. LightRAG 핵심

- **그래프 기반 인덱싱**: 평면 청크 대신 엔티티/관계 그래프로 지식 단위의 상호의존을 포착.
- **이중 레벨 검색**: 저수준(세부)+고수준(개념/관계)을 함께 검색 → 문맥성↑.
- **증분 업데이트**: 새 데이터 즉시 반영.
- **요구사항**: **임베딩 모델 + LLM(엔티티/관계 추출) + 벡터/그래프 저장소.** 그래프 유지 오버헤드.

## 2. yok3x `knot`와의 대조

| | LightRAG | yok3x knot |
|---|---|---|
| 저장 | 벡터DB + 그래프DB | **평문 md 파일**(노트당 1파일) |
| 그래프 | LLM이 엔티티/관계 자동 추출 | **`[[위키링크]]`**(수동 그래프) + 태그 |
| 검색 | 임베딩 유사도 + 그래프 탐색(이중레벨) | **키워드 스코어링**(제목3·태그2·본문1) |
| 증분 | 전용 알고리즘 | ✅ 파일 추가/수정이 곧 증분 |
| 의존성 | 임베딩·LLM·DB **다수** | **0(표준 라이브러리만)** |

## 3. 적합성 판단

### 프레임워크 통째 도입 — ✗ 부적합
- yok3x의 **의존성 0·표준 라이브러리만**(RULE) 원칙을 정면 위배. 임베딩 모델/벡터DB/그래프DB가
  들어오면 설치·런타임·외부 호출이 급증한다.
- knot의 사용 규모(런 산출물·공유 기억 md 수십~수백 개)엔 **과설계**. LightRAG는 대규모 코퍼스용.
- knot의 가치는 "가볍고 투명한 평문 지식"인데, 벡터화하면 그 장점을 잃는다.

### 아이디어 경량 이식 — ✅ 가치 있음(의존성 0 유지)
knot이 이미 **[[위키링크]]=수동 지식그래프**를 갖고 있으므로, LightRAG의 이중레벨 정신을
**표준 라이브러리만으로** 흉내 낼 수 있다:
1. **그래프 확장(저→고수준)**: `query`가 키워드로 상위 노트를 찾은 뒤(저수준), 그 노트의
   `[[링크]]`·공유 태그로 **연결 노트까지 확장**(고수준). 1~2홉.
2. **태그 클러스터(고수준)**: 태그를 개념 축으로, 질의 태그와 겹치는 노트를 가중.
3. **링크 그래프 lint 활용**: 이미 있는 `knot lint`의 링크 그래프를 검색에도 재사용.

→ 임베딩 없이 "세부(키워드) + 관계(링크/태그)" 이중 검색 근사. `knot.query`에 30~50줄 증분.

## 4. 권고

- **LightRAG 프레임워크 도입: 안 함.** 의존성 0 위배 + 과설계.
- **대안 A(권장)**: `knot.query`에 **링크/태그 그래프 확장**을 더해 이중레벨 근사(의존성 0).
  반나절 규모, knot의 투명성 유지.
- **대안 B(선택)**: 정말 대규모 외부 코퍼스 RAG가 필요하면, LightRAG를 **선택적 외부 backend**로
  (limits의 `command` probe처럼 optional). 코어엔 안 넣음.

**한 줄**: LightRAG의 "그래프+이중레벨"은 옳은 방향이고 knot의 [[링크]]와 정신이 같다. 다만
프레임워크가 아니라 **아이디어만** 가져와 knot을 경량 강화하는 게 yok3x답다.
