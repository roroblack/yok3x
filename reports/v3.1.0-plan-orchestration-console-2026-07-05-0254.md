# v3.1.0 · 오케스트레이션 콘솔 계획서 (작업 채팅 + CLI-기능 할당) · 2026-07-05 02:54

## 0. 결론 — 가능하다

yok3x 백엔드가 이미 (a) `harness gui` 로컬 서버 + `POST /api/run`, (b) config의 `workers`(역할↔backend)·`routing`(기능↔도구)·`flavors`를 갖고 있어, 그 위에 **오케스트레이션 콘솔**을 얹으면 된다. 신규 백엔드는 API 3개 + 실행 큐 정도로, 의존성 0을 유지한 채 구현 가능하다. 이 문서는 계획이며, 함께 제공하는 `v3.1.0-mockup-orchestration-console-*.html`이 예상 화면이다.

## 1. 두 기능 정의

### A. 통합 작업 채팅 콘솔 (Task Console)
- 대화창에 **수행할 작업을 각각 입력** → 각 입력이 하나의 "실행 턴"이 되어 큐에 들어가고 순차 실행.
- 입력 시 지정: 패턴(producer-reviewer / pipeline / fanout-fanin) · 이 실행에 쓸 워커(producer/reviewer 오버라이드) · 승인 모드(자동/게이트) · 선택 옵션(workdir·verify_cmd).
- 실행 중 스텝이 대화창에 실시간 카드로 흐른다: `워커 호출 → 산출 요약 → SCORE → verify 통과/실패 → 코치 경고 → 가드 정지`.
- 여러 작업을 연달아 던지면 큐에 쌓이고, 가드가 stop이면 큐가 자동 정지(폭주 방지).

### B. CLI-기능 할당 보드 (Agent Assignment Board)
- "**어떤 CLI에 어떤 기능을 맡길지**"를 화면에서 바꾼다. 개념 매핑:
  - **기능(function)** = 구현/리팩터/코드리뷰/설계검토/테스트 (routing 키)
  - **워커(worker)** = 역할 이름(claude-main 등) + **backend(CLI)** 바인딩
  - 사용자는 기능마다 **담당 CLI를 드롭다운으로 선택** → 내부적으로 워커의 backend 재바인딩 + routing 갱신.
- 3 CLI 카드(claude/codex/gemini)에 실측 한도 배지(대시보드 재활용) — "지금 여유 있는 CLI로 옮겨라"를 눈으로 보고 바로 재배치.
- flavor(오케스트레이터) 전환, 워커 역할 프롬프트 편집(고급)도 여기서. 저장 시 `yok3x.json` 반영.

## 2. 개념 모델 (기존 구조에 그대로 매핑)

```
사용자 조작                     →  config 변경
──────────────────────────────────────────────
기능 '구현'을 codex로            →  routing.build = "codex"  + (필요시) 워커 backend 재바인딩
'코드리뷰' 담당을 gemini로        →  routing.review = "gemini"
이번 실행만 producer=codex-main   →  task spec의 producer 오버라이드(저장 안 함)
오케스트레이터를 codex로          →  flavor = "codex-orchestrator"
```

- **영구 변경**(보드 저장) vs **1회 오버라이드**(채팅 입력 시)를 분리 — 실수로 전체 설정이 바뀌지 않게.

## 3. UX 흐름

1. 콘솔 탭 진입 → 좌측 채팅, 우측 할당 보드.
2. 할당 보드에서 기능↔CLI 확인/변경(예: 리뷰를 codex→gemini). "저장" 누르면 config 반영, 안 누르면 유지.
3. 채팅에 작업 입력 + 패턴 선택 + (선택)워커 오버라이드 → 전송.
4. 스텝이 대화창에 흐름 → 완료 시 최종 산출물 링크/요약. 가드 경고·정지도 대화에 표시.
5. 다음 작업 입력 → 큐잉/순차 실행.

## 4. 백엔드 추가 (guiserver 확장, 의존성 0)

| API | 역할 |
|---|---|
| `POST /api/run` (확장) | **인라인 task spec(JSON)** 수용 → `.yok3x/console/`에 임시 태스크 저장 → 백그라운드 실행(기존 `_run_task_bg` 재활용) + **실행 큐** |
| `GET /api/run/log?run_id=` | 해당 run의 `run.log` 증분 반환(대화창 스트리밍). 폴링 방식(SSE 불필요, 의존성 0) |
| `POST /api/config` | `workers[].backend` · `routing` · `flavor` · 역할 프롬프트 부분 갱신 → **검증 후** `yok3x.json` 저장 |
| `GET /api/state` (확장) | 현재 실행 큐·활성 run 요약 포함(이미 `running` 필드 있음, 큐 추가) |

- 실행 큐: 현재 `_RUN_LOCK`(단일 실행) → **대기열**로 확장. 동시 실행 방지 유지.
- 설정 저장은 원자적 쓰기(임시파일 후 교체) + 스키마 검증 실패 시 저장 거부·사유 표시(폴백으로 조용히 뭉개지 않음 — RULE 5.5).

## 5. 설계 원칙 준수

- **가드**: 채팅 실행도 매 워커 호출 전 한도 점검, stop이면 큐 정지 + 대화창 알림.
- **승인 게이트**: 채팅에 "자동/게이트" 토글. 게이트면 스텝 승인 버튼이 대화창에 뜬다(무인 폭주 방지 — 카파시 4원칙).
- **할루시네이션 방지 · 하네스 기법**: 기존 주입 그대로 적용.
- **폴백·하드코딩(RULE 5.5)**: config 편집은 검증 후 저장, 실패는 표시. 새 폴백/하드코딩 생기면 감사 리포트 갱신.
- **정직 표기**: mock 백엔드면 대화창에 `mock` 배지, 실측/추정/원장 배지 유지.

## 6. 단계별 구현 계획

- **P1 — 할당 보드(설정 편집)**: `POST /api/config` + 보드 UI. 기능↔CLI 드롭다운, flavor, 저장. (제일 독립적, 먼저)
- **P2 — 채팅 실행**: `POST /api/run` 인라인 spec + 큐 + `GET /api/run/log` 폴링. 대화형 스텝 표시.
- **P3 — 게이트/오버라이드**: 승인 게이트 웹 연동, 1회 워커 오버라이드.
- **P4 — 다듬기**: 큐 뷰·취소, 역할 프롬프트 편집, 산출물 뷰어 연결.

각 단계 Windows 검증 + HISTORY 기록 + 필요 시 감사 리포트.

## 7. 리스크 / 주의

- **웹에서 무인 실행**은 요금·폭주 위험 → 기본 게이트 모드, 자동은 명시 토글 + 경고.
- **로컬 전용**: 서버는 `127.0.0.1` 바인딩 유지(외부 노출 금지).
- 실 CLI 스트리밍 로그는 `-p` 단발이라 스텝 단위로만(토큰 스트리밍 아님) — 대화창은 스텝 카드 단위.
- config 편집 실수 방지: 저장 전 미리보기 + 되돌리기(직전 yok3x.json 백업).
