# v2.3.0 · 코딩용 적합성 점검 + 추가 기능 제안 · 2026-07-04 13:39

harness가 "코딩용 코칭 멀티에이전트 툴"로 제대로 만들어졌는지 점검하고, 더 좋은 도구가 되기 위한 기능을 우선순위로 제안한다.

---

## 1. 코딩 적합성 점검 (현재 v2.3.0)

| 항목 | 상태 | 근거 |
|---|:--:|---|
| 워커 역할이 코딩 중심 | ✅ | claude-main=구현, codex-main=리팩터, codex-critic=코드 리뷰(버그·엣지·보안), gemini=설계 검토 |
| 코드 생성↔리뷰 교차검증 | ✅ | producer-reviewer, `SCORE:` 채점, 미달 시 재작업 |
| 하네스 기법(계획→구현→검증) | ✅ | 워커 프롬프트에 자동 주입 |
| 할루시네이션 방지 | ✅ | 사실성 규칙 주입 + 검수 환각지적 + 과잉확신 체크리스트 |
| 구독 한도 준수 | ✅ | codex 라이브 실측·claude 캘리브레이트·가드 자동정지 |
| 코딩 샘플 태스크 | ✅ | 슬러그 함수·LRU 캐시·CSV 파싱 |
| 공유 코드 기억(knot) | ✅ | 런 요약·설계 노트 저장·주입 |
| **실제 코드 실행/테스트** | ❌ | 리뷰가 LLM 주관 점수뿐 — 코드를 **실행/테스트**해 객관 검증하지 않음 |
| **파일·레포에 실제 적용** | ❌ | 워커 출력이 텍스트로만 캡처됨 — 파일 쓰기/패치 적용/작업 디렉터리 없음 |
| **레포 컨텍스트 주입** | ❌ | 워커가 실제 코드베이스(관련 파일)를 못 봄 |
| **git 통합** | ❌ | 브랜치·커밋·diff 없음 |
| 스톨(수렴 실패) 감지 | ❌ | max_rounds 상한만, 같은 점수 반복 감지 없음 |

**판정**: *코칭·한도·교차검증·안전장치*는 코딩용으로 잘 갖춰졌다. 그러나 **"실제 코드를 만지고 객관적으로 검증하는"** 축이 비어 있다 — 지금은 "코드에 대해 이야기하는" 도구에 가깝고, "코드를 실제로 짜고 돌려보는" 도구까지는 한 걸음 남았다.

## 2. 추가 기능 제안 (우선순위)

### 🔴 P1 — 도구의 본질을 바꾸는 것

**1. 테스트/린트 게이트 (객관 검증)**
producer가 코드를 쓰면 `pytest`/`ruff`/`tsc`/`eslint`를 **실제 실행**해 결과를 리뷰어에 주입하고, pass/fail을 SCORE와 별개의 **하드 게이트**로 둔다. 주관 점수만으로 통과시키지 않는다. → 할루시네이션·거짓 "동작합니다"를 근본 차단.

**2. 작업 디렉터리 + 파일 적용**
워커를 실제 `cwd`에서 돌리고, claude/codex CLI가 **파일을 직접 생성·수정**하게 한 뒤 그 **diff를 캡처**한다. 지금처럼 텍스트만 받지 말고 레포에 반영. (claude/codex CLI는 원래 파일 편집 도구인데 현재는 그 능력을 안 씀.)

**3. 레포 컨텍스트 주입**
task.json에 `context_globs: ["src/**/*.py"]` 같은 필드 → 관련 파일을 프롬프트에 주입(글자 제한·요약). knot을 "코드 지식 베이스"로 확장.

### 🟡 P2 — 실용성

**4. git 통합**: 런마다 자동 브랜치, 게이트 통과 시 커밋, `diff` 표시. 실패 롤백.
**5. diff 기반 리뷰**: 리뷰어에 전체가 아니라 **변경분(diff)**만 준다 — 정확·저비용.
**6. 스톨 감지**: 직전 라운드와 점수·이슈 동일이 2회 → 조기 종료 + knot에 실패 원인 기록.
**7. rubric 파일 주입**: task.json `rubric: rubric.md` → 검수 프롬프트에 통째로. 채점 일관성.

### 🟢 P3 — 완성도

**8. 비용 인지 라우팅**: 스캐폴딩=값싼 모델, 어려운 로직·리뷰=강한 모델. 한도·비용 최적.
**9. 언어별 리뷰어 프로파일**: python/ts/rust별 체크리스트·린터 매핑.
**10. 런 재개/이력**: 중단된 런 resume, 런 이력 뷰(GUI 연동).
**11. GUI 실행 버튼 배선**: 현재 GUI의 Run/Loop 버튼을 실제 `harness run`에 연결(POST 엔드포인트).

## 3. 다음 한 걸음(추천)

**P1-1(테스트 게이트) + P1-2(파일 적용)** 을 먼저 하면, "말하는 도구"에서 "실제로 코드를 짜고 돌려 검증하는 도구"로 성격이 바뀐다. 이게 코딩용 멀티에이전트로서 가장 큰 도약이고, 할루시네이션 방지와도 직결된다(실행이 곧 검증).
