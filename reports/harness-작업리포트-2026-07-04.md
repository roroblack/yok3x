## 1. 한 줄 요약

기존 한도 "실측"은 **오래된 세션 파일 값(3주 전, 이미 리셋됨)**이라 부정확했다. 리서치로 **`codex app-server` JSON-RPC 라이브 조회**를 찾아 구현해 codex는 이제 지금-이-순간 값을 읽는다. claude·gemini는 CLI를 직접 돌려본 결과 **라이브 % 소스가 존재하지 않음**을 확인하고, 추정/원장으로 정직하게 처리했다. 동시에 워커·샘플을 코딩 워크플로우로 전환하고 Windows 크래시(BOM)를 고쳤다. 전부 실제 머신에서 검증했다.

## 2. 핵심 성과 — 실시간 한도 조회

### 2.1 문제

기존 방식은 `~/.codex/sessions/**/rollout-*.jsonl` 파일의 마지막 `rate_limits` 값을 읽었다. 그런데 그 파일이 **6월 14일** 것이라, 해당 5시간 창은 이미 리셋됐고 표시값이 현재와 무관했다.

### 2.2 리서치 결과 (도구별 실시간 실측 가능성)

| 도구 | 라이브 % 가능? | 방법 |
|---|---|---|
| **Codex** | ✅ 가능 | `codex app-server` JSON-RPC `account/rateLimits/read` (CodexBar와 동일 경로) |
| Claude | ❌ 불가 | Anthropic이 CLI로 잔여 한도 미노출. CodexBar조차 transcript 스캔으로 폴백 |
| Gemini | ❌ 불가 | 헤드리스 JSON은 호출당 토큰만, 잔여 한도 없음. Antigravity 로컬 API에만 tier 존재 |

- `codex exec --json`은 `rate_limits: null` 버그(알려진 이슈)라 사용 불가 → app-server 경로 채택.

### 2.3 구현: `codex_appserver` probe (`harness/limits.py`)

```
codex -s read-only -a untrusted app-server   (stdio JSON-RPC)
  → initialize
  → initialized
  → account/rateLimits/read
  → result.rateLimits.primary(5h)/secondary(7d).{usedPercent, windowDurationMins, resetsAt}
```

- 카멜케이스(`usedPercent`), 응답 이벤트 기반 대기, 타임아웃, Windows 프로세스 트리 kill(`taskkill /F /T`).
- 실패 시 **세션 파일(stale) → 원장** 순으로 자동 폴백.
- 조회 소요 약 1.2초, 15초 캐시로 루프 중 반복 스폰 방지.

### 2.4 검증 (같은 순간, 두 방식 비교)

| 항목 | 파일 방식(기존) | **app-server(신규)** |
|---|---|---|
| codex 7일 사용률 | 5% (stale) | **24% (live)** |
| 리셋 표시 | "리셋 임박"(오류) | **2일 13시간 후(정확)** |

**가드 정지 검증**: hard_ratio를 0.20으로 두고 실행 → codex 라이브 7일 24% > 20% → 루프 자동 STOP 확인.

## 3. Claude / Gemini — 정직한 한계

네 머신에서 CLI를 직접 실행해 확인한 사실:

- **Claude**: `claude --help`에 usage 커맨드 없음, `/usage`는 TUI 전용. → `~/.claude/projects` transcript를 5h/7d 롤링 합산해 **설정한 상한 대비 추정**. 본인 플랜의 `limit_5h_tokens`·`limit_7d_tokens`를 넣어야 활성화(0이면 원장 폴백). ccusage/CodexBar와 동일한 최선책.
- **Gemini**: `gemini -p --output-format json --skip-trust` 실행 결과 `stats.models` 토큰만 반환, 잔여 한도 없음. → 자체 일일 예산(원장)으로 준수.

mat·coach는 이를 `[실측]`·`[추정]`·`[원장]`으로 구분 표기해 오해가 없게 했다.

## 4. 코딩용 전환

- **워커 역할 재정의**: claude-main=구현, codex-main=리팩터, **codex-critic=코드 리뷰어(버그·엣지·보안·성능·가독성)**, gemini=설계 검토
- **라우팅**: `build / refactor / review / test / design_review`
- **샘플 태스크 3종 교체**:
  - `task-producer-reviewer.json`: 슬러그 변환 함수+테스트 (구현→코드리뷰→수정, max_rounds 3)
  - `task-pipeline.json`: 인메모리 LRU 캐시 (설계→구현→리뷰 파이프라인)
  - `task-fanout.json`: 대용량 CSV 스트리밍 파싱 접근법 (팬아웃→통합)

## 5. Windows 호환성

- **BOM 크래시 수정**: 모든 설정·태스크·노트 읽기를 `utf-8-sig`로 변경 → 메모장·PowerShell이 붙이는 UTF-8 BOM이 있어도 정상 로드(검증: BOM 바이트 239,187,191 파일 로드 성공).
- 기존 패치 유지·재확인: `shutil.which()`로 npm `.cmd` 심 해석, stdout/stderr UTF-8 재구성(cp949 콘솔에서 게이지 `█░╭╰` 정상).
- 한도 조회 경로는 `Path.home()` 기반(`%USERPROFILE%`).

## 6. 가드 동작 원리 (한도 준수)

1. 매 워커 호출 전 `limits.probe()`로 **실측 한도 우선** 점검.
2. `use_real_limits: true` → 실측/추정 사용률이 기준. 가장 빡빡한 창(5h/7d) 기준으로 판정.
3. `hard_ratio`(기본 1.0) 도달 → 루프 자동 정지. `soft_ratio`(0.8) → 경고.
4. `on_probe_failure`: `ledger`(기본, 원장 폴백) / `block`(fail-closed 차단) / `allow`.

## 10. 실사용 & 남은 선택

```powershell
python harness.py init
python harness.py limits    # codex 라이브 %, claude/gemini 상태 확인
python harness.py mat       # 5h/7d 게이지 + 코칭
python harness.py loop task-producer-reviewer.json -n 5   # codex 한도 넘으면 자동 정지
```

**선택 사항**: claude 추정을 켜려면 본인 Max 플랜의 5h/7d 토큰 상한을 `harness.json`의 `limits.claude.limit_5h_tokens`·`limit_7d_tokens`에 입력.
